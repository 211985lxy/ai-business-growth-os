## 二、技术选型

### 2.1 Next.js 16（前端框架）

**选择理由：**
```
✅ React 生态，灵活强大
✅ 服务端渲染（SEO 友好）
✅ API Routes（前后端一体）
✅ Vercel 部署简单
✅ App Router（最新架构）
```

**替代方案对比：**
| 方案 | 优势 | 劣势 | 选择 |
|------|------|------|------|
| Next.js | 全栈、生态 | - | ✅ |
| Vue.js | 易学 | 生态不如 React | ❌ |
| Angular | 企业级 | 学习曲线陡 | ❌ |
| 纯 React | 灵活 | 需自己搭建后端 | ❌ |

### 2.2 Dify（AI 编排）

**选择理由：**
```
✅ 可视化工作流（非技术人员可维护）
✅ 支持多种模型（灵活切换）
✅ 自托管（数据安全）
✅ 开源（可控）
✅ 中文文档（国内友好）
```

**替代方案对比：**
| 方案 | 优势 | 劣势 | 选择 |
|------|------|------|------|
| Dify | 可视化、功能全 | 学习成本 | ✅ |
| LangChain | 灵活 | 代码复杂 | ❌ |
| Flowise | 简单 | 功能不如 Dify | ❌ |
| 直接 API | 简单 | 无法编排 | ❌ |

### 2.3 硅基流动（模型网关）

**选择理由：**
```
✅ 兼容 OpenAI API（无缝切换）
✅ 价格便宜（DeepSeek-R1: ¥1/1M tokens）
✅ 国内访问快（不用翻墙）
✅ 支持多种模型（DeepSeek、Qwen、GLM）
```

**成本对比：**
| 模型 | 价格（输入/1M tokens） | 价格（输出/1M tokens） |
|------|---------------------|---------------------|
| 硅基 DeepSeek-R1 | ¥1 | ¥2 |
| OpenAI GPT-4 | ~¥18 | ~¥72 |
| 节省比例 | 95% | 97% |

### 2.4 Supabase（数据库）

**选择理由：**
```
✅ PostgreSQL（强大可靠）
✅ 自带认证（省去开发时间）
✅ 自带存储（文件上传）
✅ Row Level Security（数据安全）
✅ 实时订阅（未来功能）
```

**替代方案对比：**
| 方案 | 优势 | 劣势 | 选择 |
|------|------|------|------|
| Supabase | 全功能、开源 | - | ✅ |
| MongoDB | 灵活 | 不适合关系型 | ❌ |
| Firebase | 易用 | Vendor lock-in | ❌ |
| 自建 MySQL | - | 运维成本高 | ❌ |

### 2.5 Redis（缓存层）

**选择理由：**
```
✅ 极快（<5ms 读写）
✅ 支持过期（自动清理）
✅ Pub/Sub（消息总线）
✅ 成本低（内存便宜）
```

**使用场景：**
```
1. Swarm Memory（蜂群记忆）
   - 存储 Agent 上下文
   - TTL 2 小时自动过期

2. 消息总线
   - Agent 间通信
   - Redis Pub/Sub

3. 缓存层
   - Dify 响应缓存
   - 减少 API 调用
```

## 三、成本优化

### 3.1 短期优化（立即生效）

**策略1：Redis 缓存**

```
问题：同样的请求重复调用 Dify API

解法：缓存 Dify 响应

实现：
  1. 请求哈希作为 key
  2. Dify 响应作为 value
  3. TTL = 1 小时

效果：
  - 相同问题不重复调用
  - 节省 50%+ API 成本
```

**策略2：使用便宜的模型**

```
当前：硅基流动 DeepSeek-R1
成本：¥1 / 1M tokens（输入）
     ¥2 / 1M tokens（输出）

对比 OpenAI GPT-4：
成本：$2.5 / 1M tokens（输入）≈ ¥18
     $10 / 1M tokens（输出）≈ ¥72

节省：90%+
```

**策略3：流式输出**

```
传统：等待完整生成
  - 用户等待时间长
  - 服务器内存占用高

流式：边生成边输出
  - 用户感知快
  - 服务器内存占用低
  - 可提前终止
```

### 3.2 中期优化（需要开发）

**策略1：Dify 工作流优化**

```
问题：每次调用都走完整流程

优化：
  1. 提取公共逻辑
  2. 复用工作流片段
  3. 并行执行独立任务

效果：
  - 减少 30%+ Token 使用
```

**策略2：智能路由**

```
问题：所有请求都用同一个模型

优化：
  - 简单问题：用小模型（Qwen-7B）
  - 复杂问题：用大模型（DeepSeek-R1）
  - 预判用户意图，选择合适的 Agent

效果：
  - 减少 40%+ 模型成本
```

**策略3：用户分级**

```
免费用户：
  - 使用小模型
  - 有请求次数限制

付费用户：
  - 使用大模型
  - 无限次请求

VIP 用户：
  - 专属模型微调
  - 优先响应
```

### 3.3 长期优化（需要积累）

**策略1：自有模型微调**

```
时机：积累 10万+ 对话后

方法：
  1. 提取高质量对话
  2. 清洗、标注数据
  3. 微调 DeepSeek-V3
  4. 部署到硅基流动

效果：
  - 模型更懂商业
  - 减少 50%+ Token 使用
  - 形成数据壁垒
```

**策略2：知识蒸馏**

```
时机：自有模型上线后

方法：
  1. 大模型（Teacher）→ 小模型（Student）
  2. 小模型学习大模型的知识
  3. 保持性能，降低成本

效果：
  - 成本降低 80%
  - 性能损失 <10%
```

## 五、安全与合规

### 5.1 红蓝对抗

**常见攻击场景：**

**场景1：无限请求（DDoS）**
```
攻击：脚本每秒发送1000次请求
防御：
  1. IP 速率限制（5次/小时）
  2. IP 黑名单
  3. Cloudflare 防护
```

**场景2：注入攻击（XSS）**
```
攻击：<script>alert('XSS')</script>
防御：
  1. 输入验证
  2. React 自动转义
  3. CSP 头部
```

**场景3：Token 盗刷**
```
攻击：注册多个账号，绕过限制
防御：
  1. IP 限制（即使换账号）
  2. 邮箱验证
  3. 行为分析
```

### 5.2 超时控制

**API 调用超时：**
```typescript
const response = await fetch(url, {
  signal: AbortSignal.timeout(60000),  // 60秒
});
```

**流式响应超时：**
```typescript
const STREAM_TIMEOUT = 120000;  // 120秒
const controller = new AbortController();
setTimeout(() => controller.abort(), STREAM_TIMEOUT);
```

### 5.3 输入验证

```typescript
// 长度验证
if (niche.length < 5 || niche.length > 1000) {
  return { error: "描述长度应在5-1000字之间" };
}

// XSS 防护
const XSS_PATTERNS = [/script/i, /javascript:/i, /onerror=/i];
if (XSS_PATTERNS.some(p => p.test(content))) {
  return { error: "内容包含非法字符" };
}
```

### 5.4 优雅降级

**Redis 不可用：**
```typescript
const redisAvailable = await isRedisAvailable();
if (!redisAvailable) {
  console.warn("[Memory] Redis unavailable, proceeding without memory");
  // 继续执行，不阻塞
}
```

**Dify 超时：**
```typescript

  const response = await difyClient.createWorkflowStream(request);
  return response;
} catch (error) {
  if (error.code === "TIMEOUT") {

```
