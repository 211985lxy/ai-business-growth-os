## 一、Queen Agent

### 1.1 Queen Agent 是什么？

**类比：**
- 蜂群中的"女王蜂"
- 交响乐团的"指挥家"
- 手术室的"主刀医生"

**职责：**
- 理解用户意图
- 分解为 Agent 任务
- 协调执行顺序
- 整合输出结果

### 1.2 Queen Agent 的工作流程

```
用户输入：我想做美业培训，从0到100万

┌─────────────────────────────────────┐
│ Step 1: 理解意图                     │
│ - 用户想要创业                       │
│ - 行业是美业培训                     │
│ - 目标是从0到100万营收               │
└─────────────────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ Step 2: 任务分解                     │
│ 任务1: 天道·战略 - 能做吗？如何定位？│
│ 任务2: 地利·产品 - 卖什么产品？      │
│ 任务3: 人和·模式 - 如何设计模式？    │
│ 任务4: 神韵·内容 - 如何打造品牌？    │
│ 任务5: 财帛·转化 - 如何变现？        │
│ 任务6: 法度·风险 - 如何合规？        │
└─────────────────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ Step 3: 依赖分析                     │
│ 天道（独立）→ 可以立即执行            │
│ 地利（依赖天道）→ 等待天道完成        │
│ 人和（依赖天道+地利）→ 等待地利完成   │
│ ...                                  │
└─────────────────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ Step 4: 协调执行                     │
│ 1. 执行天道·战略                     │
│ 2. 保存结果到 Swarm Memory          │
│ 3. 执行地利·产品（注入天道结果）     │
│ 4. 保存结果到 Swarm Memory          │
│ 5. 继续执行人和、神韵...             │
└─────────────────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ Step 5: 整合输出                     │
│ - 汇总所有 Agent 的结果              │
│ - 生成完整的商业计划                 │
│ - 标注关键洞察和建议                 │
└─────────────────────────────────────┘
```

### 1.3 Queen Agent 的能力要求

**理解能力：**
- 识别用户的真实需求（不一定是字面意思）
- 识别信息缺失（需要追问）
- 识别任务复杂度（单 Agent vs 多 Agent）

**规划能力：**
- 将复杂问题拆解为子任务
- 识别任务之间的依赖关系
- 安排执行顺序（并行 vs 串行）

**协调能力：**
- 调用合适的 Agent
- 传递上下文（Swarm Memory）
- 处理 Agent 失败（重试、跳过、降级）

**整合能力：**
- 汇总多个 Agent 的输出
- 识别冲突并解决
- 生成连贯的综合报告

## 二、主动追问逻辑

### 2.1 为什么要主动追问？

**用户的问题往往是不完整的：**

```
用户问："如何提高营收？"

❌ 被动回答：列出100种方法（泛泛而谈）
✅ 主动追问：
   1. 你现在做什么业务？
   2. 当前营收是多少？
   3. 目标营收是多少？
   4. 有哪些资源？
   → 然后给出针对性的建议
```

**主动追问的价值：**
- 理解完整需求
- 避免误判
- 提供精准建议
- 形成闭环决策

### 2.2 什么时候追问？

**规则1：信息缺失时**

```
用户："我想创业"
追问：
  1. 你想进入哪个行业？
  2. 你有什么资源/经验？
  3. 你的目标是什么？
```

**规则2：信息模糊时**

```
用户："美业培训"
追问：
  1. 你是指"培训美业从业者"还是"美业技能培训"？
  2. 你的目标客户是谁？
  3. 你的差异化是什么？
```

**规则3：信息冲突时**

```
用户："我要做高端美业培训，定价99元"
追问：
  1. "高端"和"99元"似乎矛盾，你能详细说明吗？
  2. 你理解的"高端"是指什么？
  3. 为什么定价99元？
```

### 2.3 追问策略

**策略1：递进式追问**

```
第一轮：核心问题
  "你想进入哪个行业？"

第二轮：背景信息
  "你有什么相关经验？"

第三轮：资源情况
  "你有多少启动资金？"
```

**策略2：场景化追问**

```
❌ 坏的追问："请提供更多信息"
✅ 好的追问：
   1. "你现在的业务是什么？"
   2. "你希望3个月后达到什么目标？"
   3. "你遇到的最大挑战是什么？"
```

**策略3：选项式追问**

```
❌ 开放式："你的目标客户是谁？"
✅ 选项式："你的目标客户是：
    A. 想转行的美业从业者
    B. 美业店主
    C. 美业专业学生
    D. 其他（请说明）"
```

### 2.4 追问的度

**原则：**
- 不要一次性问太多（3-5个问题为宜）
- 问关键问题，不是所有问题
- 适当给出默认假设，让用户确认

```
❌ 过度追问：
   请回答以下15个问题...

✅ 适度追问：
   我理解你想做美业培训。为了给出更精准的建议，
   我需要确认2个问题：
   1. 你的目标客户是哪类人？
   2. 你的差异化是什么？
   （如果不确定，我可以基于常见情况给出建议）
```

## 三、多Agent共识

### 3.1 为什么需要共识？

**问题：** 多个 Agent 可能有不同的观点

```
场景：美业培训的定价

财帛·Agent："定价2990元，符合市场行情"
神韵·Agent："定价99元，快速获取用户"

冲突：如何定价？
```

**解决：** Queen Agent 仲裁

### 3.2 共识机制

**机制1：上下文优先**

```
规则：后面的 Agent 可以看到前面 Agent 的结论

例子：
  天道：定位为"高端创业者教育"
    ↓
  财帛：定价19800元（与定位一致）
    ↓
  神韵：品牌调性"专业、高端"（与定价一致）
```

**机制2：数据驱动**

```
规则：基于数据做决策，不是基于观点

例子：
  财帛：市场调研显示，美业培训平均定价2990元
  神韵：竞争对手分析显示，低价策略竞争激烈
    ↓
  Queen：综合判断，建议定价2990元（中端路线）
```

**机制3：用户最终决策**

```
规则：Agent 提供建议，用户做最终决策

例子：
  Queen：关于定价，有两个方案：
    方案A：2990元（中端路线，受众广）
    方案B：19800元（高端路线，门槛高）
    建议：先从方案A开始，积累案例后再升级
```

### 3.3 冲突解决流程

```
检测到冲突
  ↓
分析冲突原因
  ├─ 上下文不同 → 补充信息，重新评估
  ├─ 数据来源不同 → 标注数据差异，让用户选择
  └─ 策略偏好不同 → 提供多个方案，说明利弊
  ↓
Queen Agent 仲裁
  ↓
输出综合建议
```

## 四、通信协议

### 4.1 Swarm Protocol（蜂群通信协议）

**目标：** 让 Agent 之间可以通信

**协议格式：**

```typescript
interface AgentMessage {
  from: string;          // 发送者 Agent
  to: string;            // 接收者 Agent（或 "ALL"）
  type: "info" | "request" | "response";  // 消息类型
  data: any;             // 消息内容
  timestamp: number;     // 时间戳
  messageId: string;     // 消息ID
}
```

**使用场景：**

**场景1：信息共享**

```typescript
// 天道·战略 → 所有 Agent
{
  from: "天道·战略",
  to: "ALL",
  type: "info",
  data: {
    positioning: "美业创业者教育",
    targetAudience: "30-45岁美业从业者"
  },
  timestamp: 1709337600000,
  messageId: "msg_001"
}
```

**场景2：请求信息**

```typescript
// 财帛·转化 → 人和·模式
{
  from: "财帛·转化",
  to: "人和·模式",
  type: "request",
  data: {
    question: "目标客户的支付能力如何？"
  },
  timestamp: 1709337600000,
  messageId: "msg_002"
}
```

**场景3：响应请求**

```typescript
// 人和·模式 → 财帛·转化
{
  from: "人和·模式",
  to: "财帛·转化",
  type: "response",
  data: {
    answer: "目标客户月收入1-3万，愿意为教育付费",
    evidence: ["市场调研数据", "用户访谈"]
  },
  timestamp: 1709337660000,
  messageId: "msg_003",
  inReplyTo: "msg_002"
}
```

### 4.2 消息总线（Message Bus）

**实现：** Redis Pub/Sub

```typescript
// 发布消息
redis.publish("swarm:channel", JSON.stringify(message));

// 订阅消息
redis.subscribe("swarm:channel", (message) => {
  const msg = JSON.parse(message);
  if (msg.to === "ALL" || msg.to === currentAgent) {
    handleMessage(msg);
  }
});
```

### 4.3 Swarm Memory vs Message Bus

| | Swarm Memory | Message Bus |
|---|-------------|-------------|
| **用途** | 持久化共享上下文 | 实时通信 |
| **存储** | Redis 键值存储 | Redis Pub/Sub |
| **读取** | 按需读取 | 订阅自动推送 |
| **时效** | 长期保存（TTL 2小时） | 实时消息 |
| **例子** | "战略定位：创业者教育" | "财帛请求人和的信息" |

## 五、技术实现

### 5.1 Queen Agent 的 Dify 工作流

```
用户输入
  ↓
[LLM 节点] 理解意图
  ↓
[代码节点] 任务分解
  ↓
[条件分支] 单 Agent vs 多 Agent
  ↓
[循环节点] 依次执行 Agent 任务
  ↓
[代码节点] 更新 Swarm Memory
  ↓
[LLM 节点] 整合输出
  ↓
返回结果
```

### 5.2 主动追问的 Dify 工作流

```
用户输入
  ↓
[LLM 节点] 分析信息完整性
  ↓
[条件分支]
  ├─ 信息完整 → 继续处理
  └─ 信息缺失 → 生成追问问题
  ↓
返回追问 or 继续处理
```

### 5.3 多 Agent 共识的 Dify 工作流

```
执行 Agent A
  ↓
保存结果到 Swarm Memory
  ↓
执行 Agent B（注入 A 的结果）
  ↓
[LLM 节点] 检查冲突
  ↓
[条件分支]
  ├─ 无冲突 → 继续执行
  └─ 有冲突 → Queen 仲裁
  ↓
输出综合建议
```
