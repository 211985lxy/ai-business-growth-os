### 1.2 为什么这样选型？

**Next.js 16（前端框架）**

```
选择理由：
  ✅ React 生态，灵活强大
  ✅ 服务端渲染（SEO 友好）
  ✅ API Routes（前后端一体）
  ✅ Vercel 部署简单

替代方案：
  ❌ Vue.js：生态不如 React
  ❌ Angular：学习曲线陡
  ❌ 纯 React：需要自己搭建后端
```

**Dify（AI 编排）**

```
选择理由：
  ✅ 可视化工作流（非技术人员可维护）
  ✅ 支持多种模型（灵活切换）
  ✅ 自托管（数据安全）
  ✅ 开源（可控）

替代方案：
  ❌ LangChain：代码复杂，维护成本高
  ❌ Flowise：功能不如 Dify 完善
  ❌ 直接调用 API：无法编排复杂流程
```

**硅基流动（模型网关）**

```
选择理由：
  ✅ 兼容 OpenAI API（无缝切换）
  ✅ 价格便宜（DeepSeek-R1: ¥1/1M tokens）
  ✅ 国内访问快（不用翻墙）
  ✅ 支持多种模型

替代方案：
  ❌ OpenAI API：价格高（$5/1M tokens）
  ❌ 直连 DeepSeek：需要自己处理限流、故障
  ❌ 其他国内网关：硅基流动是性价比最高的
```

**Supabase（数据库）**

```
选择理由：
  ✅ PostgreSQL（强大可靠）
  ✅ 自带认证（省去开发时间）
  ✅ 自带存储（文件上传）
  ✅ 实时订阅（未来功能）
  ✅ Row Level Security（数据安全）

替代方案：
  ❌ MongoDB：不适合关系型数据
  ❌ Firebase：非开源，有 vendor lock-in
  ❌ 自建 MySQL：运维成本高
```

**Redis（缓存层）**

```
选择理由：
  ✅ 极快（<5ms 读写）
  ✅ 支持过期（自动清理）
  ✅ Pub/Sub（消息总线）
  ✅ 成本低（内存便宜）

替代方案：
  ❌ Memcached：功能不如 Redis
  ❌ 内存缓存：不支持分布式
  ❌ 数据库缓存：性能太差
```

## 二、成本优化

### 2.1 短期优化（立即生效）

**策略1：Redis 缓存**

```
问题：同样的请求重复调用 Dify API

解法：缓存 Dify 响应

实现：
  1. 请求哈希作为 key
  2. Dify 响应作为 value
  3. TTL = 1 小时

效果：
  - 相同问题不重复调用
  - 节省 50%+ API 成本
```

**策略2：使用便宜的模型**

```
当前：硅基流动 DeepSeek-R1
成本：¥1 / 1M tokens（输入）
     ¥2 / 1M tokens（输出）

对比 OpenAI GPT-4：
成本：$2.5 / 1M tokens（输入）≈ ¥18
     $10 / 1M tokens（输出）≈ ¥72

节省：90%+
```

**策略3：流式输出**

```
传统：等待完整生成
  - 用户等待时间长
  - 服务器内存占用高

流式：边生成边输出
  - 用户感知快
  - 服务器内存占用低
  - 可提前终止（用户看到想要的就停止）
```

### 2.2 中期优化（需要开发）

**策略1：Dify 工作流优化**

```
问题：每次调用都走完整流程

优化：
  1. 提取公共逻辑（如：行业分析）
  2. 复用工作流片段
  3. 并行执行独立任务

效果：
  - 减少 30%+ Token 使用
```

**策略2：智能路由**

```
问题：所有请求都用同一个模型

优化：
  - 简单问题：用小模型（Qwen-7B）
  - 复杂问题：用大模型（DeepSeek-R1）
  - 预判用户意图，选择合适的 Agent

效果：
  - 减少 40%+ 模型成本
```

**策略3：用户分级**

```
免费用户：
  - 使用小模型
  - 有请求次数限制

付费用户：
  - 使用大模型
  - 无限次请求

VIP 用户：
  - 专属模型微调
  - 优先响应

效果：
  - 免费用户不亏钱
  - 付费用户贡献利润
```

### 2.3 长期优化（需要积累）

**策略1：自有模型微调**

```
时机：积累 10万+ 对话后

方法：
  1. 提取高质量对话
  2. 清洗、标注数据
  3. 微调 DeepSeek-V3
  4. 部署到硅基流动

效果：
  - 模型更懂商业
  - 减少 50%+ Token 使用
  - 形成数据壁垒
```

**策略2：知识蒸馏**

```
时机：自有模型上线后

方法：
  1. 大模型（Teacher）→ 小模型（Student）
  2. 小模型学习大模型的知识
  3. 保持性能，降低成本

效果：
  - 成本降低 80%
  - 性能损失 <10%
```

**策略3：边缘计算**

```
时机：用户量大后

方法：
  1. 将简单任务部署到边缘（CDN）
  2. 复杂任务才调用中心服务

效果：
  - 降低中心服务器压力
  - 提升响应速度
```

## 三、数据资产

### 3.1 数据飞轮

```
用户使用 → 产生数据 → 优化模型 → 更好体验 → 更多用户
   ↑                                              ↓
   └──────────────────────────────────────────────┘
```

### 3.2 积累什么数据？

**对话数据：**

```sql
CREATE TABLE agent_outputs (
  id UUID PRIMARY KEY,
  user_id UUID,
  agent_type TEXT,        -- 哪个 Agent
  inputs JSONB,           -- 输入
  outputs TEXT,           -- 输出
  created_at TIMESTAMPTZ
);

价值：
  - 了解用户需求
  - 优化 Agent 提示词
  - 训练专属模型
```

**反馈数据：**

```sql
CREATE TABLE user_feedback (
  id UUID PRIMARY KEY,
  user_id UUID,
  output_id UUID,         -- 哪条输出
  feedback_type TEXT,     -- 点赞/点踩
  feedback_text TEXT,     -- 文字反馈
  created_at TIMESTAMPTZ
);

价值：
  - 了解哪些回答好
  - 了解哪些回答差
  - 持续优化
```

**行为数据：**

```sql
CREATE TABLE user_actions (
  id UUID PRIMARY KEY,
  user_id UUID,
  action_type TEXT,       -- 复制/分享/再生成
  context JSONB,          -- 上下文
  created_at TIMESTAMPTZ
);

价值：
  - 了解用户偏好
  - 优化产品功能
  - 发现新需求
```

### 3.3 数据如何"值钱"？

**短期：优化现有产品**

```
数据分析 → 发现问题 → 优化产品
  - 哪个 Agent 用得最多？
  - 哪些回答被点赞最多？
  - 用户在哪里流失？
```

**中期：训练专属模型**

```
10万+ 对话 → 微调模型 → 更懂商业
  - 比 GPT-4 更懂中国商业
  - 比 GPT-4 更懂中小企业
  - 形成数据壁垒
```

**长期：数据资产变现**

```
方式1：行业报告
  - 匿名化数据
  - 行业趋势分析
  - 定价：¥999/份

方式2：API 开放
  - 其他开发者调用
  - 按调用次数收费
  - 定价：¥0.1/次

方式3：企业版
  - 企业私有化部署
  - 使用自己的数据训练
  - 定价：¥10万/年
```

## 四、可持续运营

### 4.1 成本结构

**单次请求成本：**

```
假设：平均一次请求使用 1000 tokens（输入+输出）

成本：
  - 模型成本：¥0.002（硅基流动 DeepSeek-R1）
  - Redis 成本：¥0.0001（读写）
  - Supabase 成本：¥0.0001（存储）
  总计：¥0.0022/次

1000次请求 = ¥2.2
100万次请求 = ¥2200
```

**月度成本（1万日活）：**

```
假设：每个用户每天 10 次请求

日请求量：10万次
月请求量：300万次

成本：
  - 模型：¥6600
  - Redis：¥300
  - Supabase：¥300
  总计：¥7200/月

收入（假设 5% 付费率）：
  - 付费用户：500人
  - 月费：¥29
  - 收入：¥14500

利润：¥7300/月
```
