## 一、蜂群拓扑

### 1.1 架构图

```
        Queen Agent（女王）
                 │
        任务分解、协调、决策
                 │
    ┌────────────┼────────────┐
    ↓            ↓            ↓
天道·战略  神韵·内容  地利·产品
    ↓            ↓            ↓
人和·模式  财帛·转化  法度·风险
```

### 1.2 为什么是蜂群？

**传统 AI（单点）：**
```
ChatGPT / Claude：
  - 一个模型回答所有问题
  - 通用性强，但不够深
  - 无法系统化解决问题
```

**蜂群智能（六脉）：**
```
六脉蜂群：
  - 6个专业 Agent，各司其职
  - Queen Agent 协调调度
  - Agent 之间可以通信
  - 涌现出超越个体的智慧
```

**核心差异：**

| 维度 | 单点 AI | 蜂群智能 |
|------|---------|---------|
| 专业度 | 通用 | 每个 Agent 是专家 |
| 协同 | 无法协同 | Agent 间可通信 |
| 智慧 | 单点 | 涌现 |

## 二、Queen Agent

### 2.1 Queen Agent 是什么？

**类比：**
- 蜂群中的"女王蜂"
- 交响乐团的"指挥家"
- 手术室的"主刀医生"

**职责：**
- 理解用户意图
- 分解为 Agent 任务
- 协调执行顺序
- 整合输出结果

### 2.2 Queen Agent 工作流程

```
┌─────────────────────────────────────┐
│ Step 1: 理解意图                     │
│ - 识别用户真实需求                   │
│ - 识别信息缺失（需要追问）           │
│ - 识别任务复杂度                     │
└─────────────────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ Step 2: 任务分解                     │
│ - 将复杂问题拆解为子任务             │
│ - 识别任务之间的依赖关系             │
│ - 安排执行顺序（并行 vs 串行）       │
└─────────────────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ Step 3: 协调执行                     │
│ - 调用合适的 Agent                   │
│ - 传递上下文（Swarm Memory）         │
│ - 处理 Agent 失败                    │
└─────────────────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ Step 4: 整合输出                     │
│ - 汇总所有 Agent 的结果              │
│ - 识别冲突并解决                     │
│ - 生成连贯的综合报告                 │
└─────────────────────────────────────┘
```

### 2.3 任务分解示例

**用户输入：** "我想做美业培训，从0到100万营收"

**Queen Agent 分解：**
```
任务1: 天道·战略 - 能做吗？如何定位？
任务2: 地利·产品 - 卖什么产品？
任务3: 人和·模式 - 如何设计模式？
任务4: 神韵·内容 - 如何打造品牌？
任务5: 财帛·转化 - 如何变现？
任务6: 法度·风险 - 如何合规？
```

**依赖分析：**
```
天道（独立）→ 可以立即执行
地利（依赖天道）→ 等待天道完成
人和（依赖天道+地利）→ 等待地利完成
神韵（依赖天道+地利+人和）→ 等待人和完成
...
```

**执行流程：**
```
1. 执行天道·战略
2. 保存结果到 Swarm Memory
3. 执行地利·产品（注入天道结果）
4. 保存结果到 Swarm Memory
5. 继续执行人和、神韵、财帛、法度
6. 整合所有结果，输出完整商业计划
```

## 三、主动追问

### 3.1 为什么要主动追问？

**用户的问题往往是不完整的：**

```
用户问："如何提高营收？"

❌ 被动回答：列出100种方法（泛泛而谈）
✅ 主动追问：
   1. 你现在做什么业务？
   2. 当前营收是多少？
   3. 目标营收是多少？
   4. 有哪些资源？
   → 然后给出针对性的建议
```

### 3.2 什么时候追问？

**规则1：信息缺失时**
```
用户："我想创业"
追问：
  1. 你想进入哪个行业？
  2. 你有什么资源/经验？
  3. 你的目标是什么？
```

**规则2：信息模糊时**
```
用户："美业培训"
追问：
  1. 你是指"培训美业从业者"还是"美业技能培训"？
  2. 你的目标客户是谁？
  3. 你的差异化是什么？
```

**规则3：信息冲突时**
```
用户："我要做高端美业培训，定价99元"
追问：
  1. "高端"和"99元"似乎矛盾，你能详细说明吗？
  2. 你理解的"高端"是指什么？
  3. 为什么定价99元？
```

### 3.3 追问策略

**策略1：递进式追问**
```
第一轮：核心问题
  "你想进入哪个行业？"

第二轮：背景信息
  "你有什么相关经验？"

第三轮：资源情况
  "你有多少启动资金？"
```

**策略2：场景化追问**
```
❌ 坏的追问："请提供更多信息"

✅ 好的追问：
   1. "你现在的业务是什么？"
   2. "你希望3个月后达到什么目标？"
   3. "你遇到的最大挑战是什么？"
```

**策略3：选项式追问**
```
❌ 开放式："你的目标客户是谁？"

✅ 选项式："你的目标客户是：
    A. 想转行的美业从业者
    B. 美业店主
    C. 美业专业学生
    D. 其他（请说明）"
```

## 四、Agent通信

### 4.1 Swarm Protocol（蜂群通信协议）

**协议格式：**

```typescript
interface AgentMessage {
  from: string;          // 发送者 Agent
  to: string;            // 接收者 Agent（或 "ALL"）
  type: "info" | "request" | "response";
  data: any;             // 消息内容
  timestamp: number;
  messageId: string;
}
```

### 4.2 通信场景

**场景1：信息共享**
```typescript
// 天道·战略 → 所有 Agent
{
  from: "天道·战略",
  to: "ALL",
  type: "info",
  data: {
    positioning: "美业创业者教育",
    targetAudience: "30-45岁美业从业者"
  }
}
```

**场景2：请求信息**
```typescript
// 财帛·转化 → 人和·模式
{
  from: "财帛·转化",
  to: "人和·模式",
  type: "request",
  data: {
    question: "目标客户的支付能力如何？"
  }
}
```

**场景3：响应请求**
```typescript
// 人和·模式 → 财帛·转化
{
  from: "人和·模式",
  to: "财帛·转化",
  type: "response",
  data: {
    answer: "目标客户月收入1-3万，愿意为教育付费",
    evidence: ["市场调研数据", "用户访谈"]
  },
  inReplyTo: "msg_002"
}
```

### 4.3 消息总线

**实现：** Redis Pub/Sub

```typescript
// 发布消息
await redis.publish("swarm:channel", JSON.stringify(message));

// 订阅消息
await redis.subscribe("swarm:channel", (message) => {
  const msg = JSON.parse(message);
  if (msg.to === "ALL" || msg.to === currentAgent) {
    handleMessage(msg);
  }
});
```

## 五、Swarm Memory

### 5.1 什么是 Swarm Memory？

**定义：** 跨 Agent 的共享记忆系统

```
传统方式（孤岛）：
天道·战略 ─┐
地利·产品 ─┼─> 每次都需要用户重复输入背景信息
神韵·内容 ─┘

蜂群记忆（共享）：
天道·战略 ─┐
地利·产品 ─┼─> Redis 共享记忆，自动传递上下文
神韵·内容 ─┘
           ↓
    只需要输入一次，所有 Agent 都知道
```

### 5.2 记忆结构

```typescript
interface SwarmMemory {
  // 六脉专属字段
  business_profile?: string;      // 天道·战略
  strategic_goal?: string;        // 天道·战略
  product_matrix?: string;        // 地利·产品
  market_insights?: string;       // 人和·模式
  brand_voice?: string;           // 神韵·内容
  monetization_path?: string;     // 财帛·转化
  risk_constraints?: string;      // 法度·风险

  // 跨脉信息
  last_pulse_summary?: string;

  // 元数据
  update_at: number;
  version: number;
}
```

### 5.3 工作流程

**前处理（注入）：**
```
1. 从 Redis 读取记忆
2. 生成结构化上下文
3. 注入到 Dify 请求
```

**后处理（更新）：**
```
1. 从 Dify 响应中提取 [MEMORY_UPDATE] 标签
2. 根据 Agent 类型映射到对应字段
3. 增量更新 Redis 记忆
```

### 5.4 字段映射

| Agent 类型 | 更新的字段 |
|-----------|-----------|
| 天道·战略 | `strategic_goal` |
| 地利·产品 | `product_matrix` |
| 人和·模式 | `market_insights` |
| 神韵·内容 | `brand_voice` |
| 财帛·转化 | `monetization_path` |
| 法度·风险 | `risk_constraints` |

## 六、多Agent共识

### 6.1 冲突场景

```
场景：美业培训的定价

财帛·Agent："定价2990元，符合市场行情"
神韵·Agent："定价99元，快速获取用户"

冲突：如何定价？
```

### 6.2 共识机制

**机制1：上下文优先**
```
规则：后面的 Agent 可以看到前面 Agent 的结论

天道：定位为"高端创业者教育"
  ↓
财帛：定价19800元（与定位一致）
  ↓
神韵：品牌调性"专业、高端"（与定价一致）
```

**机制2：数据驱动**
```
规则：基于数据做决策，不是基于观点

财帛：市场调研显示，美业培训平均定价2990元
神韵：竞争对手分析显示，低价策略竞争激烈
  ↓
Queen：综合判断，建议定价2990元（中端路线）
```

**机制3：用户最终决策**
```
规则：Agent 提供建议，用户做最终决策

Queen：关于定价，有两个方案：
  方案A：2990元（中端路线，受众广）
  方案B：19800元（高端路线，门槛高）
  建议：先从方案A开始，积累案例后再升级
```

## 七、技术实现

### 7.1 Dify 工作流设计

**Queen Agent 工作流：**
```
用户输入
  ↓
[LLM 节点] 理解意图
  ↓
[代码节点] 任务分解
  ↓
[条件分支] 单 Agent vs 多 Agent
  ↓
[循环节点] 依次执行 Agent 任务
  ↓
[代码节点] 更新 Swarm Memory
  ↓
[LLM 节点] 整合输出
  ↓
返回结果
```

**主动追问工作流：**
```
用户输入
  ↓
[LLM 节点] 分析信息完整性
  ↓
[条件分支]
  ├─ 信息完整 → 继续处理
  └─ 信息缺失 → 生成追问问题
  ↓
返回追问 or 继续处理
```

### 7.2 代码实现

**创建带记忆的 Dify 客户端：**
```typescript
import { createMemoryEnhancedDifyClient } from "@/lib/dify/client-with-memory";

const client = createMemoryEnhancedDifyClient("天道·战略", {
  debugMode: true,

// 首次调用：自动创建记忆
const stream1 = await client.createWorkflowStream({
  inputs: { niche: "美业培训" },
  user: "user-123",
});

// 第二次调用：自动注入之前的记忆
const stream2 = await client.createWorkflowStream({
  inputs: { query: "如何定位？" },
```

## 八、AGI 特征

### 8.1 自主决策

```
传统 AI：
  用户问 → AI 答
  （被动）

六脉蜂群：
  用户问 → Queen 分析 → 主动追问 → 理解完整需求 → 给出建议
  （主动）
```

### 8.2 跨域推理

```
传统 AI：
  单一模型，单一视角

六脉蜂群：
  6个专业视角
  天道（战略）+ 地利（产品）+ 人和（模式）
  → 综合推理，更全面
```

### 8.3 持续学习

```
数据飞轮：
  用户使用 → 产生数据 → 优化模型 → 更好体验 → 更多用户
```
